# 開発

## 役割

技術ドキュメント、設計書、デバッグログを管理する。

## ルール

- 技術ドキュメントは `docs/topic-name.md`
- デバッグログは `debug-log/YYYY-MM-DD-issue-name.md`
- デバッグのステータス: `open → investigating → resolved → closed`
- 設計書は必ず「概要」「設計・方針」「詳細」の構成にする
- バグ修正時は「再発防止」セクションを必ず記入
- 技術的な意思決定はCEOのdecisionsにもログを残す
- **APIキーはこのファイルや.company/内に直接書かない**（.envで管理）

## Codex CLI 連携（任意）

Codex CLI（MCP経由）を使う場合、コード実装をオフロードできる。

### Codexに振るタスク

- 新規ファイルのコード実装（仕様が明確なもの）
- テストコード生成
- バグ修正（原因が特定済み）
- CSS/HTML/JSの量産・修正

### Claudeに残すタスク

- アーキテクチャ設計・技術選定
- コードレビュー（Codexの出力検証）
- セキュリティ関連のコード
- 仕様が曖昧なタスクの要件整理

### 品質保証（相互レビュー）

**鉄則: 実装者≠レビュアー。自分で書いて自分でレビューしない。**

| パターン | 実装 | レビュー | 用途 |
|---------|------|---------|------|
| A（通常） | Codex | Sonnet | コスト効率重視 |
| B（設計重視） | Sonnet | Codex | 設計判断が多い実装 |
| C（重要） | 両方 | Opus比較 | セキュリティ・決済系 |

### Codex呼び出し時の注意

- 必ず具体的な仕様を渡す（「いい感じに」はNG）
- 日本語・絵文字を含むファイルは Codex に書かせない（エンコーディングが壊れる）
- 出力は必ず別モデルがレビューしてからマージ

## 自動品質チェック（任意・CEO/秘書判断で発動）

オーナーからの指示不要。状況を見て自動で発動する。

### セキュリティ監査

- **発動条件**: 認証・決済・個人情報を扱うコード変更時、デプロイ前
- **内容**: OWASP Top 10チェック、SQLインジェクション、XSS、CSRF、認証バイパス
- **出力**: `docs/security-audit-YYYY-MM-DD.md`
- **重大な脆弱性発見時**: 即CEOにエスカレ → デプロイ停止判断

### パフォーマンス計測

- **発動条件**: LP・トップページの改修後、SEO関連の作業時
- **内容**: Core Web Vitals（LCP/FID/CLS）、ページサイズ、読み込み時間
- **出力**: `docs/benchmark-YYYY-MM-DD.md`
- **基準**: LCP 2.5秒以内、CLS 0.1以下

### ブラウザ動作テスト

- **発動条件**: UI変更・新機能追加・リリース前
- **内容**: 実際にブラウザを開いて主要導線を確認（Playwright MCP等）
- **出力**: `debug-log/YYYY-MM-DD-browser-test.md`

### 発動しないケース

- 軽微なテキスト修正・画像差し替えのみ
- .company内のドキュメント更新のみ
- 明らかにステージング/テスト環境のみの作業

## フォルダ構成

- `docs/` - 技術ドキュメント・設計書・監査レポート
- `debug-log/` - デバッグ・バグ調査ログ・テスト結果
